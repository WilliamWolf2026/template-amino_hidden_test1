### 11. ENG-1626: getUserData → PlayerIdentityService Migration

Migrated Amino's hand-rolled `getUserData()` identity resolver to game-kit's `PlayerIdentityService`.

**game-kit changes** (PR: https://github.com/wolfgames/game-kit/pull/58):
- `PlayerIdentityService.identify()` now accepts optional `IdentifyOptions` for URL param UUID resolution and legacy localStorage migration
- Fixed bug where `initIdentity()` wasn't persisting generated UUID to localStorage
- `PlayerIdentity` stays PII-free (UUID + gamerTag only) — privacy decision documented in `Lattice/docs/enhancements/pii-in-player-identity.md`

**template-amino changes** (PR: https://github.com/wolfgames/template-amino-js/pull/13):
- `setup/tracking.ts` — uses `PlayerIdentityService` for UUID, resolves email/name locally for analytics
- `setup/flags.ts` — uses `PlayerIdentityService` for UUID
- `setup/helper.ts` — **deleted** (`getUserData` was its only export)
- `src/game/INDEX.md` — removed archive dir reference, updated identity source, genericized asset example
- `docs/INDEX.md` — updated getUserData → PlayerIdentityService reference

**Privacy decision:** Storing email/name in `PlayerIdentity` at the SDK level would persist PII to localStorage in every game-kit consumer — flowing into game saves, analytics, and flag configs. Each game handles PII at its own layer instead.
